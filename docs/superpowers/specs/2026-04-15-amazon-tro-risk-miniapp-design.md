# 亚马逊美国站侵权与 TRO 风险小程序设计

## 1. 目标与结论

本项目要做的不是一个法律信息目录，而是一个面向 `Amazon 美国站卖家` 的 `知产/TRO 风险分诊与持续监控小程序`。

V1 的产品结论已经明确：

- 用户：`Amazon 美国站卖家`
- 产品形态：`工具分发型首页`
- 商业模式：`混合模式`
- 首页范围：`3 个核心工具 + 1 个顾问承接入口`
- 核心工具：`侵权体检`、`TRO 预警`、`案件进展`
- 使用方式：`无需店铺授权`
- 输入对象：`品牌词 / 店铺名 / ASIN`
- 结果策略：`免费摘要 + 留资解锁完整报告`
- 监控策略：`持续监控`
- 提醒方式：`短信 / 邮件`
- 服务承接：`有真人律师或顾问团队`

产品对外承诺是：

- 自动检索
- 风险提示
- 持续监控
- 报告解锁
- 顾问承接

产品不对外承诺：

- 自动给出正式法律意见
- 自动替代律师判断
- 覆盖所有国家和平台

## 2. 用户问题与设计原则

目标用户面临的核心问题不是“查不到信息”，而是：

1. 不知道自己是否已经接近 TRO 或侵权风险
2. 即使查到信息，也看不懂对自己有什么影响
3. 没有持续监控，通常出事后才发现

因此设计遵循四个原则：

1. 先给结果，再展开细节
2. 统一输入模型，减少学习成本
3. 风险必须附带命中原因，不能只给分数
4. 自动化负责检索、归纳、提醒，不冒充正式法律结论

## 3. V1 范围

### 范围内

- 美国范围
- Amazon 美国站卖家
- 输入 `品牌词 / 店铺名 / ASIN`
- 三个核心工具：
  - `侵权体检`
  - `TRO 预警`
  - `案件进展`
- 持续监控任务
- 免费摘要
- 留资后查看完整报告
- 短信 / 邮件提醒
- 顾问承接入口

### 范围外

- 非美国市场
- Walmart、TikTok Shop、独立站等多平台支持
- 卖家账号授权后的深度店铺数据
- 自动生成正式律师意见
- 复杂版权实时主链路

## 4. 首页信息架构

首页不是六宫格功能目录，而是 `风险分诊台`。

目标：

1. 用户在 10 秒内选对入口
2. 一次性查询自然转为持续监控
3. 首页本身体现长期价值，而不是一次性落地页

建议结构：

1. 顶部公告区
- 活动、重要提醒、服务通知

2. 统一查询区
- 输入框支持：`品牌词 / 店铺名 / ASIN`
- 主 CTA：`立即检测`
- 系统自动识别输入类型，不要求用户手动选择

3. 核心工具区
- `侵权体检`：查商标/专利/版权高风险点
- `TRO 预警`：监控新案，提前发现冻结风险
- `案件进展`：追踪涉案案件的最新动态
- `真人顾问`：作为服务兜底入口，不与主工具竞争首屏焦点

4. 我的监控区
- 展示用户最近监控的品牌词、店铺名、ASIN
- 标出最新状态，例如 `监控中`、`2 条新变更`、`高风险提醒`

5. 报告与服务区
- 展示最近报告摘要或高价值案例摘要
- CTA：`解锁完整报告`

6. 底部 Tab
- `首页`
- `监控`
- `报告`
- `消息`
- `我的`

## 5. 三个核心工具的用户流程

三类工具共享同一套交互原则：

- 首页统一输入对象
- 自动识别输入类型
- 统一的结果页骨架
- 统一支持“加入监控”和“联系顾问”
- 对不适合直接计算的输入，允许先做一次目标确认或选择

### 5.1 侵权体检

输入：

- ASIN
- 商品链接
- 品牌词
- 店铺名

流程：

1. 拉取 Amazon 商品基础信息
2. 匹配 USPTO 商标与辅助权利信号
3. 输出免费摘要
4. 用户留资后解锁完整报告
5. 可继续创建持续监控任务

说明：

- 商品链接在进入任务前先标准化为 ASIN
- 如果用户从首页输入店铺名，系统先返回该店铺下的代表商品列表，由用户选择具体商品后再进入体检

摘要必须展示：

- 总体风险等级
- 主要风险点
- 建议动作

完整报告补充：

- 命中的商标/疑似权利基础
- 命中字段与证据
- 建议动作清单

### 5.2 TRO 预警

输入：

- 品牌词
- 店铺名
- ASIN

流程：

1. 检索联邦法院新案及相关 docket
2. 判断是否命中新案或相关主体
3. 输出免费摘要
4. 用户留资后解锁完整报告
5. 引导创建持续监控任务

摘要必须展示：

- 最近是否出现相关新案
- 命中对象
- 风险变化

完整报告补充：

- 案件列表
- 原告/律所/法院/时间
- 与监控对象的关联原因

### 5.3 案件进展

输入：

- 案件号
- 品牌词
- 店铺名
- ASIN

流程：

1. 匹配已有案件
2. 拉取最新 docket 与关键节点
3. 输出免费摘要
4. 用户留资后解锁完整报告
5. 可加入持续跟踪

摘要必须展示：

- 当前案件阶段
- 最近一次关键进展
- 对卖家的直接影响

完整报告补充：

- 时间线
- 关键文书节点
- 下一步可能动作

## 6. 统一结果页结构

结果页统一为 5 个区域：

1. 摘要卡
- 风险等级
- 一句话结论
- 最近更新时间

2. 关键证据
- 最多先展示 3 条关键证据
- 每条说明命中对象、命中字段、命中时间、关联逻辑

3. 建议动作
- 例如 `继续观察`
- `立即复核 Listing`
- `准备申诉材料`
- `联系顾问`

4. 完整报告解锁区
- 免费摘要默认可见
- 完整报告需用户留手机号或邮箱后解锁
- 解锁后立即展示，不依赖人工回传

5. 监控与顾问区
- 主 CTA：`加入监控`
- 次 CTA：`联系顾问`

## 7. 后台系统架构

后台按职责拆分为六层：

### 7.1 输入与任务接口

负责：

- 接收小程序输入
- 识别输入类型
- 创建一次性查询任务
- 创建持续监控任务

### 7.2 数据源连接器

每个数据源独立封装，不把业务规则写进连接器。

V1 需要三类连接器：

- Amazon 商品信息连接器
- USPTO 商标连接器
- CourtListener / PACER 法院数据连接器

### 7.3 归一化与实体关系

负责：

- 品牌词、原告、权利人、店铺名、ASIN 的归一化
- 别名聚合
- 去重
- 模糊匹配
- 跨源关联

### 7.4 风险引擎

负责输出：

- 风险等级
- 命中原因
- 建议动作
- 置信度

### 7.5 报告服务

负责把评分结果转成：

- 免费摘要
- 完整报告
- 短信内容
- 邮件内容

### 7.6 通知与 CRM 对接

负责：

- 短信提醒
- 邮件提醒
- 留资同步
- 顾问工作台或 CRM 同步

## 8. 数据源策略

### 8.1 Amazon 商品信息

V1 不要求用户授权 Amazon 店铺，因此不能把 seller 授权接口作为主路径。

设计要求：

- Amazon 数据连接器必须支持替换
- V1 先接入公开商品信息提取或第三方公开商品目录
- 以后再补卖家授权增强版连接器

### 8.2 USPTO 商标

USPTO 商标是 V1 的主权利信号之一。

用于：

- 商标状态查询
- 权利人信息查询
- 相关文档查询

### 8.3 CourtListener / PACER

法院数据是 `TRO 预警` 与 `案件进展` 的核心数据源。

设计要求：

- 检索接口负责发现案件与命中对象
- PACER / RECAP 补充 docket、entries、parties、documents
- 对未归档材料允许异步补抓
- 产品文案不能承诺所有案件都即时完整返回

### 8.4 版权信号

版权不作为 V1 主判断轴，只作为辅助信号。

原因：

- 实时性较差
- 在线判定成本高
- 对首版核心价值不是最短路径

## 9. 风险分级与自动化边界

### 9.1 风险分级

统一使用四档：

- `clear`：未发现明显风险
- `watch`：建议关注
- `suspected_high`：高风险疑似
- `confirmed`：明确命中

### 9.2 自动化边界

系统可以自动完成：

- 输入识别
- 数据抓取
- 实体归一
- 风险评分
- 摘要生成
- 报告生成
- 持续监控
- 短信/邮件提醒

系统不应伪装自动完成：

- 正式法律意见
- 和解/申诉策略的最终结论
- 高争议命中的最终裁定

### 9.3 产品文案边界

首页、结果页、报告页都要明确：

`本工具提供信息检索、风险提示与持续监控，不构成正式法律意见。`

## 10. 异常处理与误判控制

### 10.1 数据不足时不硬判

如果商品信息不全、商标信息不全、法院材料未归档：

- 不输出“低风险”
- 输出“信息不足，暂不能下结论”
- 提供补充输入或后台补抓指引

### 10.2 高争议命中先标疑似

对于主体近似、品牌词近似、关联链不完整的情况：

- 先归为 `watch` 或 `suspected_high`
- 不直接标记为 `confirmed`

### 10.3 所有结论必须附原因

结果页与报告页必须说明：

- 命中了什么
- 为什么命中
- 命中时间
- 关联逻辑

### 10.4 误判控制原则

- 规则优先，模型辅助
- 证据先于结论
- 只有 `suspected_high` 与已留资用户进入人工复核池

## 11. 测试策略

测试分四层：

1. 连接器测试
- 验证抓取字段、超时、限流、缺失字段处理

2. 归一化与风险引擎测试
- 使用固定样本回归
- 覆盖明确命中、明确不命中、模糊命中、数据不足

3. 报告生成测试
- 确保页面摘要、完整报告、短信、邮件的一致性

4. 端到端流程测试
- 从首页输入、报告解锁、创建监控任务到通知触发的全链路测试

## 12. 实现顺序

V1 按下面顺序实施：

1. 统一输入与任务骨架
2. TRO 预警
3. 案件进展
4. 侵权体检
5. 完整报告解锁
6. 短信 / 邮件提醒
7. 顾问承接与 CRM 同步

原因：

- `TRO 预警` 最容易体现持续监控价值
- `案件进展` 可复用法院数据层
- `侵权体检` 复杂度最高，放后面更稳

## 13. 成功标准

V1 至少应满足：

1. 用户可在首页输入 `品牌词 / 店铺名 / ASIN`
2. 三个工具都能返回统一摘要结构
3. 用户可留资后查看完整报告
4. 用户可创建持续监控任务
5. 系统可通过短信或邮件发送命中提醒
6. 结果页的每个结论都有证据说明
